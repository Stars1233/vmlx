# vMLX v1.6.0 Autodetect + Toggles + Defaults Audit

Audit date: 2026-06-30. Author: Jinho Jang. Source: pre-release code review
before Codex UI validation on erics-m5-max.local.

## Scope

Every autodetect path, per-family toggle, and startup default that changes
behavior in v1.6.0 vs v1.5.69 must be verified live in the dev-build UI
before shipping. This document maps the code paths so Codex can drive the
matrix.

## 1. Model-family autodetect

**Primary entry:** `panel/src/main/model-config-registry.ts:1002`
`detectModelConfigFromDir(modelPath)` — reads `config.json`.

Detection order:
1. `config.json.model_type` → `MODEL_TYPE_TO_FAMILY` map (`:290`)
2. JANG Tier-1 fallback via `jang_config.capabilities.family` (`:1042`)
3. Name-regex fallback (GLM-Z1 disambig `:1049`, MedGemma `:1054`)

Ornith uses `qwen3.5` / `qwen3.5-moe` families (`:121-122`); VL detection
defers to `config.json.vision_config`, NOT `isMultimodal`.

Engine re-detection: `vmlx_engine/cli.py:914` calls
`get_model_config_registry().lookup(model_path)`. Manual override:
`--model-family` (`:571-587`).

**UI verify:** session log line
`[SESSION] Model family: X | tool: Y | reasoning: Z | autoTool: ... | VLM: ...`
at `sessions.ts:2979`.

**SUSPECT:** `qwen3-vl-moe` maps to `qwen3-vl` family (`:301`) but
Qwen3.5-MoE VL bundles hit `qwen3_5_moe_text` inner mapping — VL detection
then relies on `vision_config` presence. Confirm Ornith VL still triggers
`VMLX_QWEN_VL`.

## 2. Cache defaults (Phase-2 v6)

Version const: `panel/src/main/sessions.ts:571`
`CACHE_STACK_STARTUP_DEFAULTS_VERSION = 6`. Migration:
`applyCacheStackStartupDefaults` `:648-763`.

| Family                           | paged | block-disk | notes                            |
| -------------------------------- | ----- | ---------- | -------------------------------- |
| DSV4                             | ON    | ON         | 256-tok blocks, gated on opt-in  |
| MiniMax-M3                       | OFF   | OFF        | legacy disk_cache ON, JIT OFF    |
| ZAYA CCA                         | ON    | ON         | maxCacheBlocks=1000              |
| Generic / Gemma-SWA / MoE / hybrid | ON  | ON         | blockDiskCacheMaxGb=10           |

Stale-v3-v5 migration: `stalePhase1GenericPagedOff` fingerprint
(`:704-711`) forces paged-ON on upgrade — highest-impact user-visible flip
for anyone updating from v1.5.61.

**UI verify:** open session settings → "Use Paged Cache" ✅ +
"Block Disk Cache" ✅ pre-checked for gemma4/qwen3.5/generic; DSV4 shows
256-tok blocks; M3 shows paged OFF + legacy disk ON. Startup log:
`Paged cache capacity: ...` (`:634`).

`--enable-block-disk-cache` (`cli.py:2504`) requires `--use-paged-cache`;
`cli.py:1364` disables + WARNs if paged is off.

## 3. JIT auto-on for JANG-affine

**Panel:** `sessions.ts:3137`
`effectiveEnableJit = enableJit && !isVLM && !flashMoe && !distributed &&
                     !dsv4Active && !m3Active && !zayaCcaActive &&
                     !turboQuantActive && !hybridCacheActive`.

**Engine autodefault:** `cli.py:927-965` — no `--enable-jit` +
`jang_config.format ∈ {affine, jang, jjqf, mxq}` +
family ∉ {deepseek_v4, minimax_m3, minimax_m3_vl} → sets
`args.enable_jit=True`, logs
`"JANG affine model detected (format=X, family=Y) — defaulting --enable-jit ON..."`.
Kill switch: `VMLX_DISABLE_JANG_AFFINE_JIT_DEFAULT=1`.

Hard-off (override user): DSV4 `cli.py:349-353`, M3 `cli.py:533-540`.
Distributed mutex `:1309`.

**UI verify:** session log carries either
`"JANG affine model detected ... defaulting --enable-jit ON"` or
`"MiniMax-M3 detected — ignoring --enable-jit"`. `/health` returns
`_enable_jit`.

**SUSPECT:** panel excludes `turboQuantActive` but engine autodefaults ON
for JANG-affine. For JANG-affine + TurboQuant bundles the panel checkbox
may show OFF while engine actually runs JIT-ON. Contract mismatch;
consider aligning panel to mirror engine autodefault.

## 5. Reasoning-runaway backstop

All in `vmlx_engine/server.py`:
- Chat Completions non-stream `:12215-12280`
- Chat Completions stream `:15108-15170`
- Responses non-stream `:14353`
- Responses stream `:15129, :15158`
- Anthropic `:16449-16495`
- Ollama delegates through chat-completions

**Trigger:** empty content + non-empty reasoning + `enable_thinking not False` +
family ∈ {qwen3_5, qwen3_5_moe, gemma4, minimax_m3, minimax_m3_vl}
(`:12241-12242`). Re-runs with `enable_thinking=False`,
`max(32, max_tokens or 256)` budget.

No dedicated env var — family whitelist + per-request flag. M3-specific
`VMLX_M3_THINK_BUDGET` is separate.

**UI verify:** ask Ornith / Qwen3.5-MoE a thinking-heavy question that
runs the reasoning block dry; confirm final answer appears + engine log
`"%s non-stream chat produced no visible content; running bounded thinking-off answer pass"`.

## 6. VMLX_QWEN_VL toggle (Ornith VL)

Panel `sessions.ts:1624-1626` sets env `VMLX_QWEN_VL='1'` when
`freshDetectedFamily ∈ {'qwen3_5', 'qwen3_5_moe'}`.

Engine reads inside qwen35_vl patch suite —
`_patch_attention_text_rope + GatedDelta` — enables `_has_media` VL path.

**UI verify:** load Ornith VL bundle → send image → confirm decode.
Non-VL Qwen3.5 bundles no-op via `_has_media` check.

## 7. MRoPE none-delta patch

`vmlx_engine/utils/mlx_vlm_compat.py:63-72` —
`_patch_qwen35_language_mrope_none_delta()`. Default ON. Kill switch:
`VMLX_DISABLE_QWEN35_MROPE_PATCH=1`.

Without patch, `get_rope_index` runs per decode step when
`rope_deltas is None` — 10 tok/s on 397B Ornith vs 20-30 tok/s expected.

**UI verify:** load Ornith 397B → measure decode t/s. A/B via env var.
